We are moving to an electronic and cloud-based lab notebook. Zach found a solution that uses Google Drive for storage, StackEdit for editing, and GitHub for publishing. Zach has the framework in place.

As a new user getting started, the procedure goes like this:

  1. This assumes that you have logged out of Google and GitHub, and have cleared browser cookies.
  2. Navigate to stackedit.io
  3. (Optional) In the left pane window, select all and delete to create a clean document. Insert some custom text (e.g. “Hello World!”).
  4. In the upper right is a file name (e.g. “Hello!”); click on it to select it; delete it and replace it with a custom name (e.g. “StackEditTest”).
  5. # Syncronize Save on Google Drive
    -This will save “StackEditTest” to your Google Drive account and keep it syncronized. Click OK
    -Redirection: You are being redirected to Google authorization page. Click OK
  6. Log in to Google with lab credentials.
    -“StackEditTest” is now synced to Google Drive.
  7. # Publish GitHub
    -Repository: LettSpinorLab/LabNotebook.git
    -Branch: gh-pages
    -File path: StackEditTest.html
    -Format: Template
    -Redirection: You are being redirected to GitHub authorization page. Click OK
  8. Log in to GitHub with lab credentials.
    -“StackEditTest.html” is now published on GitHub.
  9. To view rendered doc, go to: LettSpinorLab.github.io/LabNotebook/StackEditTest.html

If a lab notebook file has already been created, synced, and published using the above steps, but you are using a new computer, follow these steps:

  1. Navigate to stackedit.io
  2. # Syncronize Open from Google Drive
    -Redirection: You are being redirected to Google authorization page. Click OK
  3. Log in to Google with lab credentials.
  4. Select the desired document.
  5. Edit document.

When working with a lab notebook in a browser that is logged in to the lab Drive and GitHub accounts, in StackEdit:
-To manual sync to Drive: Sync button (upper right)
-To publish to GitHub: Upload/Push button (upper right)

A list of useful StackEdit symbols/icons can be found here:
https://stackedit.io/res/libs/fontello/demo.html

2/2/2016

I have put together a mean-field truncated-Wigner (TWA) simulation in Mathematica based on these dynamical equations from Hamley’s thesis:

To check that the spinor oscillation code works, I’ve compared the rho0 and theta (spinor phase) data with Arne’s Mathematica notebook “delme.nb” for various parameters:






In the above plots, Arne’s calculations are in blue, mine are dotted orange. Aside from a Mod[pi] issue with calculating the phase (apparently due to a branch point), the two sets of calculations match up exactly. As a result, I’m confident that there are not any errors in this simulation code for a pure state.

The essential code (should be copyable into Mathematica):
(note energies are in units of angular frequency)

dynEqs[c_, E1_, E0_, 
  En1_, \[Zeta]1init_, \[Zeta]0init_, \[Zeta]n1init_] :=
 {
  \[Zeta]1'[
    t] == -I (E1*\[Zeta]1[t] + 
      c*(\[Zeta]1[t]*Conjugate[\[Zeta]1[t]] + \[Zeta]0[t]*
          Conjugate[\[Zeta]0[t]] - \[Zeta]n1[t]*
          Conjugate[\[Zeta]n1[t]]) \[Zeta]1[t] + 
      c*\[Zeta]0[t]^2 Conjugate[\[Zeta]n1[t]]),
  \[Zeta]0'[
    t] == -I (E0*\[Zeta]0[t] + 
      c*(\[Zeta]1[t]*Conjugate[\[Zeta]1[t]] + \[Zeta]n1[t]*
          Conjugate[\[Zeta]n1[t]]) \[Zeta]0[t] + 
      2 c*\[Zeta]1[t] *\[Zeta]n1[t] *Conjugate[\[Zeta]0[t]]),
  \[Zeta]n1'[
    t] == -I (En1*\[Zeta]n1[t] + 
      c*(\[Zeta]n1[t]*Conjugate[\[Zeta]n1[t]] + \[Zeta]0[t]*
          Conjugate[\[Zeta]0[t]] - \[Zeta]1[t]*
          Conjugate[\[Zeta]1[t]]) \[Zeta]n1[t] + 
      c*\[Zeta]0[t]^2 Conjugate[\[Zeta]1[t]]),
  \[Zeta]1[0] == \[Zeta]1init,
  \[Zeta]0[0] == \[Zeta]0init,
  \[Zeta]n1[0] == \[Zeta]n1init
  }
\[Zeta]1initFrom\[Rho]0[\[Rho]0_, m_, \[Chi]1_] := 
 E^(I*\[Chi]1) Sqrt[(1 - \[Rho]0 + m)/2]
\[Zeta]0initFrom\[Rho]0[\[Rho]0_] := Sqrt[\[Rho]0]
\[Zeta]n1initFrom\[Rho]0[\[Rho]0_, m_, \[Chi]n1_] := 
 E^(I*\[Chi]n1) Sqrt[(1 - \[Rho]0 - m)/2]
en1z = sim[2 \[Pi] (qhz[277]/qt[10^-4]^2)*(bf)^2 + 2 \[Pi] q\[Mu]w];
en0z = 0;
enn1z = sim[2 \[Pi] (qhz[277]/qt[10^-4]^2)*(bf)^2 + 2 \[Pi] q\[Mu]w];
\[Zeta]1initz = \[Zeta]1initFrom\[Rho]0[\[Rho]0init, m, chipinit];
\[Zeta]0initz = \[Zeta]0initFrom\[Rho]0[\[Rho]0init];
\[Zeta]n1initz = \[Zeta]n1initFrom\[Rho]0[\[Rho]0init, m, chiminit];
sols = NDSolve[
  dynEqs[c, en1z, en0z, 
   enn1z, \[Zeta]1initz, \[Zeta]0initz, \[Zeta]n1initz], {\[Zeta]1, \
\[Zeta]0, \[Zeta]n1}, {t, 0, 0.1}]

For simulating the case where rho0 = 1 and we have parametric amplification of quantum fluctuations in the stretched states, this code is run for a set of {rho0, m} parameters generated from randomly selected values of {Lx, Ly, Nyz, Nxz} from normal distributions of width N. Means and standard deviations are then extracted from the set of results.

The essential code for generating the parameter distributions is:

randOp[] := RandomVariate[NormalDistribution[0, 1/Sqrt[numAtoms]]]
tw = Table[
   lx = randOp[]; ly = randOp[]; nyz = randOp[]; nxz = randOp[];
   argP = -((ly + nyz)/(lx + nxz));
   argN = (ly - nyz)/(lx - nxz);

   chiP = If[lx + nxz > 0, ArcTan[argP], ArcTan[argP] + \[Pi]];
   chiM = If[lx - nxz > 0, ArcTan[argN], ArcTan[argN] + \[Pi]];
   rho0 = 
    1/2 + Sqrt[
     1/4 - 1/8 (((lx + nxz)/Cos[chiP])^2 + ((lx - nxz)/ Cos[chiM])^2)];
   mm = 1/(
     8 rho0) (((lx + nxz)/Cos[chiP])^2 - ((lx - nxz)/Cos[chiM])^2);
   {chiP, chiM, rho0, mm},
   {i, numSamp}];

Some thougts on purchasing:

Microwave source

Looking at a replacement signal generator for the microwaves. Potential problems with the current system:

  1. HP 8657B is old, and might not perform as well (and will eventually stop working).
  2. The mixing process produces many other signals at the wrong frequency. Impedance matching (and waveguide physics) probably prevents much of it from making it into the chamber, but a non-mixing method or a filter might be a good idea. (Some research on filters showed that a custom filter might be the way to go; stock filters tend to be too wide-band for this application.)
  3. We don’t know what the phase-noise and spectral purity of the FPGA RF source is.

I’ve looked at two alternatives so far:

  1. BNC 835-M
  2. AnaPico APSIN20G-1URM

I’m considering options for replacing our present microwave source. Currently, we mix the outputs of an HP 8657B (fixed at 1.74 GHz, +13 dBm) with an FPGA RF source (at 30 MHz) using a signal-sideband modulator (Polyphase) to produce 1.772 GHz, which is then amplified and sent to an antenna in our experiment. We control the system output power and frequency by controlling the FPGA. The application here is driving transitions between the hyperfine ground states of sodium (1.771626128 GHz).

I would either like to replace the HP 8657B (datasheet attached) with a newer signal generator (and continue to do the mixing), or replace the whole system with a single fast-switching, programmable, low-noise, 1.772 GHz source.
